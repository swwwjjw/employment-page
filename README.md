## SRS — Спецификация требований к ПО
**Проект:** `employment-page` — статическая страница прeбординга ("Шаги трудоустройства") для новых сотрудников аэропорта Пулково  
**Версия документа:** 1.0  
**Дата последнего изменения:** 22.12.2025

### Введение

#### Назначение
Данный документ описывает требования к веб‑странице «Шаги трудоустройства», предназначенной для информирования кандидатов/новых сотрудников о необходимых этапах оформления и предоставления ссылок на инструкции, формы и справочные материалы.

#### Соглашения документа
- Требования формулируются как **«Система должна…»**.
- Приоритеты: **P1 (критично)**, **P2 (важно)**, **P3 (желательно)**.
- Идентификаторы требований: **FR-*** (функциональные), **NFR-*** (нефункциональные), **OR-*** (другие).

#### Целевая аудитория документа
- **Заказчик/HR**: чтение разделов «Product features», «System features», «User interfaces».
- **Разработчики/DevOps**: «Operating environment», «External interface requirements», «Constraints», «Security requirements».
- **Эксплуатация**: «Non functional requirements», «Assumptions and dependencies», «User documentation».

#### Область проекта
Система представляет собой **одну статическую HTML‑страницу** с встроенными стилями/скриптами и набором локальных ассетов (шрифты, изображения, PDF), раздаваемых через **nginx** в контейнере Docker.  
Система **не содержит серверной логики**, **не хранит персональные данные**, использует **внешние ссылки** на обучающие страницы/формы/справочные материалы.

#### Ссылки и артефакты
- `README.md` — SRS, инструкция по локальному запуску проекта.
- `employment-steps.html` — основная страница.
- `Dockerfile`, `compose.yaml` — сборка/запуск контейнера.
- `docker/nginx/default.conf` — конфигурация nginx (index, cache, security headers).
- `docs/DOCKER_TRANSFER_PLAN.md` — план упаковки/переноса/развертывания.
- Внешние ресурсы (пример): ссылки на обучающие инструктажи, Telegram/формы/карты (встроены в HTML).

### Описание проекта

#### Контекст продукта
Продукт — самостоятельная статическая веб‑страница, размещаемая в контейнере nginx и доступная по HTTP. Используется как информационный «путеводитель» по этапам трудоустройства и как точка входа к внешним материалам (инструктажи, формы, боты, PDF).

#### Основные возможности
- Выбор подразделения/службы из выпадающего списка.
- Отображение контента по этапам (01…06) с учетом выбранной службы.
- Ссылки на PDF‑инструкции и внешние ресурсы (обучение, формы, Telegram-бот/чаты, карты).
- Возможность отметить этап как выполненный (локально в UI на текущей странице).

#### Классы пользователей
- **Новый сотрудник/кандидат**: просматривает шаги, открывает ссылки, сверяет чек‑листы.
- **HR**: проверяет актуальность контента, дает ссылку кандидатам, обновляет артефакты через пересборку.
- **DevOps/администратор**: разворачивает/обновляет контейнер, контролирует доступность.

#### Операционная среда
- **Клиент**: современный браузер (Chrome/Edge/Firefox/Safari), экран мобильный/десктоп.
- **Сервер**: Docker Engine (24+ желательно), контейнер `nginx:alpine`, HTTP порт 80 (с маппингом, например 7000:80).
- **Сеть**: доступ к внешним доменам по HTTPS для перехода по ссылкам.

#### Ограничения дизайна и реализации
- Страница и ассеты **встроены в образ**; Изменения контента требуют **пересборки** образа (**Constraint C1**).
- Нет БД/бэкенда; Функциональность ограничена клиентским JavaScript (**C2**).
- Заголовки безопасности и политика контента задаются nginx (**C3**, см. CSP в конфигурации).
- Ошибки 404 перенаправляются на `employment-steps.html` (поведение задано конфигом nginx) (**C4**).

#### Пользовательская документация
- Для эксплуатации/локального запуска: `README.md`, `docs/DOCKER_TRANSFER_PLAN.md`.
- Для конечного пользователя: содержимое `employment-steps.html` (инструкции на странице и ссылки на PDF/внешние ресурсы).

#### Допущения и зависимости
- Внешние ссылки (инструктажи, формы, Telegram, карты) доступны и не требуют нестандартной аутентификации (**A1**).
- PDF/изображения, включенные в образ, являются актуальными версиями на момент сборки (**A2**).
- Пользователь использует браузер с включенным JavaScript (иначе динамическая подстановка контента ограничена) (**A3**).
- TLS/реверс‑прокси (если нужен) настраивается **вне** контейнера (**A4**).

### Системные функции

#### Системная функция 1: Просмотр и навигация по шагам трудоустройства (P1)

##### Описание
Система должна предоставлять пользователю доступ к последовательности этапов трудоустройства и содержимому каждого этапа.

##### Триггер-действие и ожидаемая реакция
- **Триггер:** пользователь открывает страницу.  
  **Реакция:** отображаются заголовок, селектор службы, блоки этапов; контент этапов заполняется для службы по умолчанию.
- **Триггер:** пользователь прокручивает страницу.  
  **Реакция:** контент остается доступным; отображение корректно на мобильных/десктопных размерах.

##### Функциональные требования
- **FR-1:** Система должна отображать список этапов трудоустройства с уникальными заголовками и порядковыми номерами.
- **FR-2:** Система должна отображать контент каждого этапа в читаемом виде (текст, списки, изображения).
- **FR-3:** Система должна предоставлять ссылки на локальные документы (PDF) и открывать их по клику.

#### Системная функция 2: Выбор службы и динамическая подстановка контента (P1)

##### Описание
Система должна позволять выбрать службу/подразделение и показывать релевантные инструкции, списки документов и ссылки для выбранной службы.

##### Триггер-действие и ожидаемая реакция
- **Триггер:** пользователь изменяет значение в выпадающем списке «Выберите службу».  
  **Реакция:** система обновляет содержимое этапов, показывая «Информация для: <выбранная служба>» и соответствующие шаблоны.

##### Функциональные требования
- **FR-4:** Система должна предоставлять элемент управления выбора службы (dropdown).
- **FR-5:** Система должна обновлять контент затронутых этапов при смене службы без перезагрузки страницы.
- **FR-6:** Система должна отображать явный контекст выбранной службы в каждом динамическом блоке.
- **FR-7:** Если для этапа/службы нет специфичного шаблона, система должна показывать базовый контент (или контекстный блок без деталей).

#### Системная функция 3: Переходы на внешние ресурсы (обучение/формы/боты) (P2)

##### Описание
Система должна предоставлять пользователю переходы на внешние ресурсы для прохождения инструктажей, заполнения форм и отправки документов.

##### Триггер-действие и ожидаемая реакция
- **Триггер:** пользователь нажимает кнопку/ссылку на внешний ресурс.  
  **Реакция:** открывается новая вкладка/окно (если предусмотрено атрибутами ссылки), без потери текущей страницы.

##### Функциональные требования
- **FR-8:** Система должна предоставлять кликабельные ссылки/кнопки на внешние ресурсы, указанные в контенте.
- **FR-9:** Внешние ссылки должны открываться с безопасными атрибутами (например, `rel="noopener noreferrer"`) при `target="_blank"`.

#### System feature 4: Отметка этапа как выполненного (локально в UI) (P3)

##### Описание
Система должна позволять пользователю отметить этап как выполненный в интерфейсе для удобства прохождения.

##### Триггер-действие и ожидаемая реакция
- **Триггер:** пользователь нажимает кнопку «Отметить этап…».  
  **Реакция:** этап визуально помечается как выполненный, текст кнопки меняется на подтверждающий; повторное нажатие отменяет отметку.

##### Функциональные требования
- **FR-10:** Система должна предоставлять кнопку отметки выполнения для каждого этапа.
- **FR-11:** Система должна менять визуальное состояние этапа и подпись кнопки при переключении состояния.
- **FR-12:** Отметка выполнения не должна требовать серверного хранения и действует в рамках текущей сессии страницы (если не реализовано иное).

### Требования к внешнему интерфейсу

#### Пользовательские интерфейсы
- Адаптивная HTML‑страница на русском языке.
- Основные UI‑элементы:
  - Заголовок страницы.
  - Dropdown выбора службы.
  - Карточки/секции этапов с заголовками и контентом.
  - Кнопки/ссылки на внешние материалы.
  - Кнопка «этап выполнен» (переключатель состояния).

#### Программные интерфейсы
- **HTTP сервер:** nginx раздает статические файлы из `/usr/share/nginx/html`.
- **Контейнеризация:** Docker (образ на базе `nginx:alpine`).
- **Зависимость от браузера:** выполнение встроенного JavaScript для динамической подстановки контента.

#### Аппаратные интерфейсы
Специальных аппаратных интерфейсов нет. Требуется устройство пользователя (ПК/смартфон) с браузером и доступом в сеть.

#### Коммуникационные интерфейсы
- Протокол: **HTTP/1.1** (внутри/на уровне контейнера).  
- Внешний доступ может быть через reverse proxy/TLS termination (вне данного продукта).
- Переходы по внешним ссылкам: **HTTPS** к сторонним ресурсам.

### Нефункциональные требования

#### Производительность
- **NFR-1:** Страница должна быть доступна к первому отображению (FCP) в пределах разумного для статического контента; целевой ориентир: до 2 секунд на типичном корпоративном канале.
- **NFR-2:** Динамическое обновление контента при смене службы должно происходить без заметной задержки (ориентир: < 200 мс на типичном устройстве).

#### Безопасность/безвредность по содержанию
- **NFR-3:** Контент (инструкции, чек‑листы, ссылки) должен быть актуальным и проверенным ответственным лицом перед публикацией (организационное требование).
- **NFR-4:** Страница не должна побуждать пользователя вводить персональные данные непосредственно в этой системе (т.к. нет механизмов хранения/защиты в продукте).

#### Атрибуты качества
- **NFR-5 (Надежность):** Контейнер должен считаться «здоровым», если `employment-steps.html` доступен по HTTP (healthcheck).
- **NFR-6 (Сопровождаемость):** Обновление контента должно выполняться заменой статических файлов и пересборкой образа.
- **NFR-7 (Совместимость):** UI должен корректно работать в актуальных версиях основных браузеров.
- **NFR-8 (Доступность):** Элементы управления должны иметь базовые атрибуты доступности (например, `aria-label`/`aria-pressed` для кнопок/селектора), а контент оставаться читаемым на мобильных экранах.

#### Требования безопасности
- **NFR-9:** Сервер должен выставлять защитные заголовки (например, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Content-Security-Policy`).
- **NFR-10:** Продукт не должен собирать/хранить персональные данные пользователей (в рамках текущей архитектуры).
- **NFR-11:** Внешние ссылки, открываемые в новой вкладке, должны использовать `rel="noopener noreferrer"`.

### Прочие требования
- **OR-1 (Развертывание):** Должна быть возможность локального запуска через Docker Compose и серверного запуска через Docker (см. `compose.yaml`, `docs/DOCKER_TRANSFER_PLAN.md`).
- **OR-2 (Кэширование):** Шрифты и изображения должны кешироваться браузером с длительными сроками (на уровне nginx), чтобы снизить нагрузку и ускорить повторные открытия.
- **OR-3 (Логи/диагностика):** Ошибки доступности (например, отсутствие файла) должны быть диагностируемы стандартными средствами контейнера (логи nginx/healthcheck).
